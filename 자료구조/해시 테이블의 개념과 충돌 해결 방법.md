# 해시 테이블
- 키(Key)를 해시 함수로 변환하여, 그 결과를 인덱스로 사용하여 값을 저장 / 조회하는 자료 구조
	- 해시 함수: 데이터를 효율적으로 관리하기 위해 데이터를 수학적 연산을 통해 고정된 길이의 데이터로 매핑하는 함수
- 해시 테이블 동작 원리
	- 해시 함수로 Key를 인덱스로 변환 (항상 같은 키는 같은 hash값을 가짐)
	- hash 값을 배열 인덱스로 사용
	- 충돌: 서로 다른 key가 같은 index를 가리킬 경우
		- 체이닝(Chaining): 같은 인덱스에 연결 리스트 저장
		- 개방 주소법(Open Addressing): 빈 공간을 찾아 이동 및 저장
- 장점
	- 매우 빠른 탐색 속도
	- 필요한 만큼의 메모리만을 사용하여 데이터를 저장하기 때문에 공간 효율성이 높음
- 단점
	- 충돌 발생 가능 -> 성능 저하
	- 순서 보장 불가능

## 충돌 해결 방법
### Chaining
중복된 해시 값이 있는 경우, 해당 슬롯을 연결 리스트로 저장.
- 최악의 경우 수행 시간이 O(n)

### Open Addressing
충돌이 발생하면 다른 빈 슬롯을 찾아 저장하는 방식. 모든 데이터는 해시 테이블 배열 안에 저장.
- 하나의 슬롯에 하나의 데이터만 저장
- 삭제 처리 복잡

선형 조사(Linear Probing): 충돌 발생 시 다음 칸을 순차적 탐색
- 구현이 매우 간단
- 캐시 친화적
- 1차 군집화(Primary Clustering, 특정 구간에 데이터가 몰림)현상 발생

이차 조사(Quadratic Probing): 탐색 간격을 제곱으로 증가
- 선형 조사보다 군집화 감소
- 2차 군집화(Secondary Clustering) 존재
- 테이블 크기 설계가 까다로움

이중 해시(Double hashing): 두번째 해시 함수를 사용하여 탐색 간격 결정
- 군집화 현상이 가장 적음
- 성능이 가장 안정적
- 해시 함수 2개 필요
- 구현 복잡도 증가

### 재해싱(Rehashing)
해시테이블의 부하율이 일정 수준을 넘으면 테이블 크기를 키우고 모든 데이터를 다시 해싱
- 성능 유지
- 충돌 빈도 감소
- 재해싱 시 비용이 큼

### Swift에서 충돌 해결 방법
- 개방 주소법 기반의 해시 테이블 사용하며, 충돌 시 선형 조사를 하고 성능 균형을 위해 Robin Hood Hashing과 재해싱 활용
	- 삭제 시 tomstone으로 삭제 마커를 표시 후 선형 조사 시 tomstone을 만나도 계속 탐색
	- Robbin Hood Hashing: 선형 조사(Linear Probing)에서 테이블의 각 키에 대해 원래 위치와 실제 위치간의 거리 (해당 키를 조회하기 위해 필요한 탐색 수)를 함께 저장하는 방법
